# Chess-style ranked Wordle system contract — Wave P

Date: 2026-07-09
Owner: Elisa
Ticket: 110 — Chess-Style Ranked System Contract
Status: architecture/contract only; no code implemented

## 1. Executive contract

Wordle Royale should model ranked play more like chess.com/lichess:

- players choose a **format/time-control**;
- players choose **ranked vs unranked** where the format supports it;
- ranked 1v1 is the core loop;
- lobbies can be casual/social or rated competitive;
- rating and stats are tracked **separately per competitive mode**;
- server authority owns pairing, puzzle selection, timing, result adjudication, and rating events.

### MVP ranked modes

Use these public mode ids for Wave P contracts:

| Public mode id | Product label | Players | Rated? | Core scoring rule |
|---|---|---:|---|---|
| `standard_1v1` | Standard | 2 | yes | Fewer guesses wins; same guesses draw. |
| `speed_1v1` | Speed / Blitz | 2 | yes | Fewer guesses wins; same guesses faster solve wins. |
| `classic_1v1` | Classic | 2 | yes | Lower time pressure; fewer guesses wins; same guesses draw. |
| `multiplayer_lobby` | Multiplayer / Lobby | 2–4 MVP, expandable later | yes or unranked | Placement-based; pairwise conversion for rating when rated. |
| `casual_lobby` | Casual Lobby | 2–4 MVP | no | Same gameplay, no rating changes. |

Implementation note: current contracts have `gameModes = ['standard']` and Prisma `MatchMode = casual|ranked`. Wave P should add an explicit **format/time-control dimension** instead of overloading `MatchMode`. Keep `MatchMode` or replacement field for rated category (`ranked`/`casual`), and add a separate `rankedMode` / `ruleset` / `timeControl` field for `standard_1v1`, `speed_1v1`, `classic_1v1`, and `multiplayer_lobby`.

## 2. Product rules — user-facing

### 2.1 Ranked 1v1 match rules

Every ranked 1v1 match must be generated by the server:

1. Both players receive the same puzzle and ruleset.
2. Server chooses dictionary release, answer, start time, end time, max guesses, and time control.
3. Clients submit guesses; server validates dictionary legality, attempt number, timing, and idempotency.
4. Clients never submit final result, rating delta, placement, answer hash/salt, or opponent outcome.

#### Standard 1v1

| Situation | Result |
|---|---|
| One player solves, other fails | Solver wins. |
| Both solve with different guess counts | Fewer guesses wins. |
| Both solve with same guess count | Draw. |
| Both fail | Draw. |
| One abandons after reveal/start | Abandoning player loses. |
| Disconnect before reveal/start | No-contest unless abuse heuristics say otherwise. |

Recommended initial config:

```text
rankedMode=standard_1v1
players=2
wordLength=5
maxGuesses=6
rounds=1 initially; can later support best-of-N
roundTimeSeconds=180 or existing 120 if product wants faster preview
sameGuessTieBreaker=draw
```

#### Speed / Blitz 1v1

| Situation | Result |
|---|---|
| One player solves, other fails | Solver wins. |
| Both solve with different guess counts | Fewer guesses wins. |
| Both solve with same guess count | Faster solve wins, using server-received timestamps. |
| Same guesses and same server solve time bucket | Draw. |
| Both fail | Draw for MVP, unless Ashar approves progress/time tiebreak. |
| Abandon after reveal/start | Abandoning player loses. |

Recommended initial config:

```text
rankedMode=speed_1v1
players=2
wordLength=5
maxGuesses=6
rounds=1
roundTimeSeconds=60-90, final value requires Ashar approval
sameGuessTieBreaker=server_solve_time_ms
```

Speed timing must use server timestamps, not client-only timestamps. Client timestamps may be stored for UX/debugging but cannot adjudicate ranked results.

#### Classic 1v1

Classic is the slower, less time-pressure ladder.

| Situation | Result |
|---|---|
| One player solves, other fails | Solver wins. |
| Both solve with different guess counts | Fewer guesses wins. |
| Both solve with same guess count | Draw. |
| Both fail | Draw. |
| Abandon after reveal/start | Abandoning player loses. |

Recommended initial config:

```text
rankedMode=classic_1v1
players=2
wordLength=5
maxGuesses=6
rounds=1 initially; likely best-of-N later
roundTimeSeconds=300 or async-friendly later; final value requires Ashar approval
sameGuessTieBreaker=draw
```

### 2.2 Ranked vs unranked lobby semantics

| Lobby type | Public/private | Rating changes | Matchmaking | Allowed in MVP |
|---|---|---|---|---|
| Quick ranked 1v1 queue | system queue | yes | server queue | yes |
| Public ranked lobby | public | yes | lobby join/start | yes, if anti-abuse constraints are met |
| Private ranked lobby | private/invite | no for MVP | invite/social | **disabled for MVP** unless Ashar approves abuse risk |
| Public unranked lobby | public | no | lobby browser | yes |
| Private unranked lobby | private/invite/code | no | invite/social | yes |

Rules:

- `rated=true` must require server-supported ranked mode and ranked-compatible lobby settings.
- Unranked games can use the same puzzle/gameplay engine but must not create `RatingEvent` rows.
- Ranked lobbies with manually chosen opponents are more abuse-prone than automatic queue matches. MVP should prefer queue-based ranked 1v1; ranked public lobbies are acceptable only with audit logging and repeat-opponent safeguards.
- Private rated lobbies remain disabled for MVP.

### 2.3 Multiplayer / lobby ranked rules

For rated multiplayer lobbies, use placement-based pairwise conversion for MVP:

- 1st place beats every lower placement.
- 2nd loses to 1st and beats lower placements.
- Equal placement groups count as draws.
- DNF/abandon after reveal/start places below all active finishers unless the match is voided.
- Scale total rating delta down as lobby size grows so one 4-player game does not swing ratings like three full independent 1v1 matches.

MVP formula shape for Ruby to simulate:

```text
for each participant A:
  pairwiseResults = compare A to each other participant by placement group
  rawDelta = sum(pairwise expected-score delta)
  scaledDelta = rawDelta * multiplayerScale(playerCount)
  cappedDelta = clamp(scaledDelta, -maxDelta, +maxDelta)
```

Suggested scale factors for simulation, not locked:

```text
2 players: 1.00
3 players: 0.75
4 players: 0.60
```

## 3. Rating model contract

### 3.1 User-facing vs internal rating

User-facing product language should stay simple:

- show a single rating number per mode;
- show provisional status;
- show games played, W/L/D, streak, recent delta;
- show graph/history per mode.

Internal implementation should preserve migration room:

- start with Elo-compatible MVP if faster;
- design schema to support Glicko-style confidence/deviation and later volatility;
- do not expose internal RD/volatility unless intentionally designed later.

### 3.2 Recommended path

Prefer **Glicko-style internal model with simple displayed rating** if implementation cost is acceptable.

Fallback acceptable for Wave P implementation:

- Elo-compatible rating number;
- per-mode profiles;
- provisional/high-K behavior;
- fields reserved for `ratingDeviation`, `volatility`, and `lastRatedAt`.

Current schema already has `RatingProfile.rating`, `matchesPlayed`, `provisionalRemaining`, `algorithm`, `algorithmConfigVersion`, and `RatingEvent`. It lacks per-mode format granularity and Glicko/confidence fields.

### 3.3 Per-mode rating profile shape

Required read model for each mode:

```ts
type RankedMode = 'standard_1v1' | 'speed_1v1' | 'classic_1v1' | 'multiplayer_lobby';
type RatingStatus = 'active' | 'suspended' | 'reset';

type ModeRatingProfile = {
  userId: string;
  mode: RankedMode;
  displayName: string;
  handle: string | null;
  avatarUrl: string | null;

  rating: number;              // displayed integer rating
  ratingDeviation?: number;    // internal confidence; omit from UI unless needed
  volatility?: number;         // internal, future Glicko support
  provisional: boolean;
  provisionalGamesPlayed: number;
  provisionalGamesTotal: number; // MVP: 10
  gamesPlayed: number;

  wins: number;
  losses: number;
  draws: number;
  abandons: number;
  currentStreak: { kind: 'win' | 'loss' | 'draw' | 'none'; count: number };

  lastRatingDelta: number | null;
  peakRating: number | null;
  rank: number | null;
  percentile: number | null;

  algorithm: string;              // e.g. elo_provisional_v1 or glicko_wordle_v1
  algorithmConfigVersion: string; // e.g. ranked_modes_v1
  status: RatingStatus;
  lastRatedAt: string | null;
};
```

### 3.4 Rating event shape

Every rating change must be replayable/auditable:

```ts
type RatingEvent = {
  eventId: string;
  ratingProfileId: string;
  userId: string;
  matchId: string;
  participantId: string;
  mode: RankedMode;
  type: 'apply' | 'void' | 'reversal' | 'adjustment';
  status: 'applied' | 'voided' | 'reversed';

  ratingBefore: number;
  ratingAfter: number;
  delta: number;
  ratingDeviationBefore?: number;
  ratingDeviationAfter?: number;
  volatilityBefore?: number;
  volatilityAfter?: number;

  outcome: 'win' | 'loss' | 'draw' | 'abandon' | 'void';
  algorithm: string;
  algorithmConfigVersion: string;
  idempotencyKey: string;
  createdAt: string;

  metadata: {
    opponentUserIds: string[];
    expectedScore?: number;
    actualScore?: number;
    provisionalBefore: boolean;
    provisionalAfter: boolean;
    placement?: number;
    placementGroup?: number;
    lobbySize?: number;
    reason?: string;
  };
};
```

## 4. Matchmaking queue contract

### 4.1 Queue request

Endpoint implication:

```text
POST /matchmaking/tickets
DELETE /matchmaking/tickets/{ticketId}
GET /matchmaking/tickets/{ticketId}
```

Request DTO:

```ts
type CreateMatchmakingTicketRequest = {
  clientRequestId: string;
  mode: 'standard_1v1' | 'speed_1v1' | 'classic_1v1';
  rated: true;
  allowProvisionalOpponent?: boolean; // default true
  regionHint?: string;                // future; not authoritative
};
```

Response DTO:

```ts
type MatchmakingTicketDto = {
  ticketId: string;
  state: 'queued' | 'matched' | 'cancelled' | 'timed_out' | 'failed';
  mode: RankedMode;
  rated: boolean;
  ratingAtQueue: number;
  searchWindow: { minRating: number; maxRating: number; expansionStep: number };
  estimatedWaitSeconds: number | null;
  matchedMatchId: string | null;
  createdAt: string;
  expiresAt: string;
};
```

Current `quickJoinRequestSchema` has `rated`, `mode`, `difficulty`. It should evolve to explicit matchmaking ticket semantics and mode ids above.

### 4.2 Search expansion

MVP expansion schedule:

| Elapsed | Rating window |
|---:|---:|
| 0s | ±100 |
| 10s | ±200 |
| 20s | ±300 |
| 30s+ | ±400, then offer unranked/lobby fallback if no match |

Rules:

- Match within the same `RankedMode` only.
- Prefer provisional vs provisional when both are available.
- Avoid immediate repeat opponents within a configurable cooldown.
- Do not pair suspended/reset profiles.
- Speed mode should eventually consider latency/region; for MVP, record region/latency hints but do not overfit.

### 4.3 Queue storage implication

MVP can use database-backed queue rows if traffic is small. For robust scale later, move hot queue coordination to Redis or a durable queue service.

Schema implication:

```text
MatchmakingTicket
- id
- userId
- mode
- rated
- state
- ratingAtQueue
- searchMinRating
- searchMaxRating
- expansionStep
- matchedMatchId nullable
- createdAt / updatedAt / expiresAt / cancelledAt
- idempotencyKey unique
```

Redis remains omitted initially for preview. Do not introduce Redis in Wave P unless a later ticket explicitly approves it.

## 6. Data model implications

### 6.1 Existing schema gaps

Current Prisma observations:

- `MatchMode` is only `casual|ranked`, which is category not format.
- `RatingProfile.mode` currently uses `MatchMode`, so it cannot represent Standard/Speed/Classic/Multiplayer separately.
- `LeaderboardSnapshot.mode` also uses `MatchMode`, so it cannot snapshot per chess-style ladder.
- `RatingProfile` lacks W/L/D counters, Glicko/RD fields, peak/rank support, and last rated timestamp.
- Match participants store `outcome`, `placement`, `finalScore`, but not first-class guesses-used/solve-time/result-group fields.

### 6.2 Recommended schema direction

Do not implement in Ticket 110, but Freya should consider:

```prisma
enum RankedMode {
  standard_1v1
  speed_1v1
  classic_1v1
  multiplayer_lobby
}

enum MatchRatingCategory {
  casual
  ranked
}
```

Add fields:

```text
Lobby.ratedCategory or mode category: casual|ranked
Lobby.rankedMode nullable RankedMode
Lobby.settings Json remains for versioned config, but key fields should be queryable if needed

Match.ratingCategory casual|ranked
Match.rankedMode nullable RankedMode
Match.rulesetVersion string

RatingProfile.mode RankedMode
RatingProfile.ratingDeviation nullable Int/Float
RatingProfile.volatility nullable Float
RatingProfile.wins/losses/draws/abandons Int defaults
RatingProfile.peakRating nullable Int
RatingProfile.lastRatedAt nullable DateTime

RatingEvent.mode RankedMode
RatingEvent.metadata includes expectedScore/actualScore/opponents/provisional/ruleset
LeaderboardSnapshot.mode RankedMode

MatchmakingTicket table as above if queue work starts
```

Migration compatibility:

- Existing `mode='ranked'` rating profiles should map to `standard_1v1`.
- Existing leaderboard `mode='ranked'` should map to `standard_1v1`.
- Existing casual lobbies/matches remain casual/unrated.
- Do not delete old data in preview; migrate or backfill deterministically.

## 7. Server-authority and anti-abuse constraints

Non-negotiable constraints:

1. Server selects puzzle, dictionary release, start/end times, and scoring/tiebreak rules.
2. Server validates every guess and stores accepted attempts idempotently.
3. Server computes standings and rating events exactly once per completed ranked match.
4. Rating event idempotency key must include match id, algorithm config version, participant profile, and event type.
5. Rating events are append-only; corrections use `void`/`reversal`/`adjustment`, not mutation.
6. Private rated lobbies stay disabled for MVP.
7. Repeat opponent cooldown should exist for ranked queue and public rated lobbies.
8. Disconnect before reveal/start is no-contest unless abuse signals trigger penalty.
9. Disconnect/abandon after reveal/start is a loss for the abandoning participant.
10. Speed tiebreaks use server-received timestamps; client timestamps are advisory only.
11. Spoiler safety remains intact: no answer word, answer hash salt, or unrevealed solution leaks in public DTOs.
12. Admin/manual adjustments require audit logs with reason and actor.

Anti-abuse events to audit:

- repeated aborts before reveal;
- repeated disconnects after seeing puzzle;
- repeated matches against same account/IP/device cluster;
- extreme rating transfer patterns in public lobbies;
- suspicious timing/client timestamp deltas;
- failed idempotency/replay attempts.

## 8. Decisions requiring Ashar approval

These are product decisions, not purely technical decisions:

1. Final public labels: `Speed` vs `Blitz`, and whether `Classic` is timed, slower timed, or async-like.
2. Exact time controls: Speed seconds, Standard seconds, Classic seconds/async behavior.
3. Whether Speed uses solve-time tiebreak for same guesses from day one. Elisa recommends yes.
4. Whether both-fail Speed games are draws or use progress/time tiebreak. Elisa recommends draw for MVP.
5. Whether public rated lobbies are enabled in MVP or only automatic ranked queue. Elisa recommends queue-first; public rated lobby behind safeguards.
6. Whether private rated lobbies remain disabled. Elisa recommends disabled for MVP.
7. Rating algorithm path after Ruby simulation: Glicko-style now vs Elo-compatible with migration fields.
8. Starting rating and provisional duration. Elisa recommends 1500 and 10 games unless Ruby simulation says otherwise.
9. Whether multiplayer/lobby ranked launches in the same wave as 1v1 or remains read-model/UI preparation only.
10. Whether inactive players get confidence decay in the first implementation.
